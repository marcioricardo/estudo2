GIT - Guia para Consulta Rápida:

Iniciando o Git

Dentro da pasta a ser usada executar git ini


Usar o comando git add + nomedoarquivo


Depois usar git commit -m + "Descrição do que foi alterado entre aspas"


Para enviar as alterações ao GitHub git push origin master

Onde "origin" refere-se ao GitHub e "master" é o nosso repositório local.

Igualando o Repositório Local ao GitHub

É preciso linkar a branch local com a brach remota, para que o GitHub entenda que estou adicionando conteúdo a uma branch que ele já possui. Por isso preciso de um comando que diga que eu estou lhe enviando uma nova branch que foi criada em meu repositório local.

git push -u origin design Para uma branch com nome "design".

Agora posso usar git pulldentro de minha brach local para trazer consteúdo que possa ter sido incluído no GitHub por outro usuário



Git branch -r Exibe as branchs remotas que estão mapeadas.

Se usar git pull podemos ver se há alguma branch remota que ainda não foi mapeada.



Se eu quiser incluir em meu repositório local uma branch que está no GitHub o comando será:
git branch -t design origin/design Com isso crio uma nova branch chamada design que já estará linkada com a branch remota de mesmo nome.



Agora posso usar inclusive git log para ver as alterações realizadas por outros usuários.

Merge Automático

Para evitar conflitos o Git pode executar um "Merge" no arquivo. Nesse caso ele exibe uma mensagem avisando que isso ocorreu. Caso não consiga ele exibe um erro. Nesse caso será preciso fazer um git pull e corrigir o arquivo manualmente.

Após o arquivo corrigido, fazer um git push Para enviar ao GitHub o arquivo corrigido.

Boas Práticas Para Evitar Conflitos:

É preciso seguir um fluxo de trabalho bem defnido afim de evitar conflitos registrados no log.

Para isso preciso sempre checar se há alguma alteração no arquivo que eu estou modificando e caso haja realizar um "merge" local. Enviando somente após isso o arquivo atualizado ao GitHub.


O processo para que isso aconteça é o seguinte:


Criar uma branch local para trabalhar. Não é recomendado trabalhar direto na branch "Master".

git checkout -b desenvolvimento Cria uma branch chamada desenvolvimento e já muda direto pra ela.

Torna-se necessário então transportar as alterações reapzadas na branch "desenvolvimento" para a branch "Master".

O primeiro passo é checar se a branch "Master" local que está conectada a branch remota, possui nova atuapzação. Alguma modificação feita enquanto trabalhávamos localmente.

Executar então: git checkout master para mudar para a branch "Master" e depois
git pull para buscar novas alterações feitas por outras pessoas.

Se não ocorreram novas alterações no arquivo remoto, executar git merge. Dessa forma eu sincronizo o conteúdo da branch local "desenvolvimento" com a branch "Master" que via de regra é a branch usada por outros usuários remotos.

git push Para sincronizar branch "Master" local com a branch "Master" remota.

Evitando Merges Automáticos

Se for verificado que há novas alterações na branch "Master" que não usamos como base para o denvolvimento de nosso arquivo, teremos um problema a resolver.

É nesse momento que caso simplesmente joguemos nosso conteúdo no "Master" ocorrerão merges automáticos. É isso que queremos evitar.

Devemos então usar o comando git rebase. Dessa forma poderemos indicar uma nova base de commits e resolver os possíveis conflitos commit por commit.


Para realizar esse procedimento devemos estar então na branch desenvolvimento.

git checkout desenvolvimento. E informar que a nova base dela é o último commit realizado na branch "Master". git rebase master desenvolvimento

Feito isso, os commits realizados em desenvolvimento são movidos para uma branch temporária e o git atualiza a nova base de commits. Após essa atualização, o próprio Git traz de volta os commits que realizamos e os aplica sobre a nova base, um de cada vez.

Nesse momento, todos os commits estão aplicados e organizados na branch "desenvolvimento".


Agora, é necessário levar esses commits para a branch "master". Repare que é um processo diferente do rebase, onde apenas queríamos trocar a base de commits utilizada. Agora, é necessário colocar apenas os commits novos na branch "master", porém, para isso, existe o comando
git merge que move os commits de uma determinada branch para outra branch.

Para utilizá-lo, primeiro é preciso ir para a branch para a qual se quer levar os commits, no caso, a "master" git checkout master.

Em seguida, deve-se dizer para o comando git merge de qual branch virão os commits novos, que, no caso, é a branch "desenvolvimento". Para isso, basta executar git merge desenvolvimento.

Nesse instante, tem-se todos os novos commits na branch "master", prontos para serem enviados para o repositório remoto através do comando git push.

Controle Avançado de Alterações

Caso eu tenha salvo um arquivo sem ter adicionado para commit ou realizado commit e quiser retroná-lo ao estado do último commit realizado, devo usar o comando:
git checkout exemplo.html
Onde "exemplo.html" é o nome do arquivo que desejo reverter ao estado do último commit.
